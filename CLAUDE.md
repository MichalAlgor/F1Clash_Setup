## Project Overview

F1 Clash Setup Manager — a web app for managing car part inventory, building setups, and comparing configurations for the F1 Clash mobile game. Built with Rust, server-rendered HTML, and htmx for interactivity.

## Tech Stack

- **Backend:** Rust + Axum
- **Templating:** Maud (compile-time HTML macros)
- **Interactivity:** htmx (HTML attributes, no JS framework)
- **Styling:** PicoCSS (classless CSS, loaded via CDN)
- **Database:** Neon (serverless PostgreSQL) via sqlx
- **Hosting:** Render.com (Dockerfile-based deploy)

## Commands

```bash
cargo check          # Type-check without building
cargo build          # Build debug binary
cargo run            # Run dev server (needs DATABASE_URL in .env)
cargo build --release # Build release binary
```

SQLx uses compile-time query checking. For CI/Docker builds without a database:
```bash
cargo sqlx prepare                    # Generate offline query data (requires running DB)
SQLX_OFFLINE=true cargo build         # Build using cached query data
```

## Architecture

```
src/
  main.rs            # Axum server setup, DB pool, router composition
  models/            # Domain types + sqlx FromRow derivations
    part.rs          # Part, PartCategory (enum), Stats
    setup.rs         # Setup (7 part references), SetupWithStats
  routes/            # Axum handlers (one file per resource)
    parts.rs         # CRUD for parts inventory
    setups.rs        # CRUD for setups + stats computation
  templates/         # Maud HTML templates
    layout.rs        # Base page shell (nav, head, picocss/htmx CDN links)
    parts.rs         # Parts list + form pages
    setups.rs        # Setups list + form pages
migrations/          # SQLx migrations (run automatically on startup)
static/              # Static assets served at /static
```

## Key Patterns

- **Server-side rendering:** All HTML is generated by Maud templates. No client-side JS framework — htmx handles dynamic updates via HTML attributes (`hx-delete`, `hx-swap`, etc.).
- **State = PgPool:** Axum router uses `PgPool` as shared state, injected via `State(pool)` extractor.
- **Part categories:** 7 fixed categories (Engine, FrontWing, RearWing, Sidepod, Underbody, Suspension, Brakes) stored as a PostgreSQL enum `part_category`.
- **Stats:** Each part has 5 stats (speed, cornering, power_unit, reliability, pit_stop). Setup stats are the sum of its 7 parts.
- **Migrations:** Auto-run on startup via `sqlx::migrate!()`. Place new migrations in `migrations/` with incrementing numeric prefixes.

## Environment

Copy `.env.example` to `.env` and set `DATABASE_URL` to your Neon connection string.
